# スマホ実機検証マニュアル（Climo）

> 対象：チームメンバー全員 / 最終更新：2026-06-28
> 目的：同じ Wi-Fi につないだ**スマホ実機**でアプリを開き、表示・ログイン・コーデ提案まで確認する。
> 所要：初回 10〜15 分。2 回目以降は数分。

このマニュアルは「PC では動くのにスマホで開けない」をゼロから解消できるよう、**順番どおりにやれば必ず動く**形でまとめています。専門用語は最小限です。

> 📌 **本マニュアルがスマホ実機検証の手順の正本（single source of truth）**です。
> [`docs/setup.md`](./setup.md) 側は要点と導線のみを置き、詳細手順は本マニュアルに集約します
> （二重管理による乖離を防ぐため）。

---

## 0. 前提（最初に一度だけ）

- PC で `docker compose up` してアプリが動いている（<http://localhost:3000> が開ける）。
- スマホと PC を**まったく同じ Wi-Fi**につなぐ。
  - ⚠ 自宅でも「ゲスト用 SSID」は端末同士の通信が遮断されることがある。**メインの SSID** を使う。
  - ⚠ スマホは**モバイル回線（4G/5G）では不可**。必ず Wi-Fi。

---

## 1. PC の LAN IP（プライベート IP）を調べる

Windows の PowerShell で次を実行：

```powershell
Get-NetIPAddress -AddressFamily IPv4 |
  Where-Object { $_.IPAddress -match '^(192\.168\.|10\.|172\.(1[6-9]|2[0-9]|3[01])\.)' -and $_.InterfaceAlias -notmatch 'Loopback|vEthernet|WSL' } |
  Select-Object IPAddress, InterfaceAlias
```

`Wi-Fi` の行の `IPAddress`（例：`192.168.128.103`）を控える。これが**あなたの PC の住所**です。
以降このマニュアルでは `<PC_IP>` と書きます（自分の値に読み替えてください）。

> 簡易版でよければ `ipconfig` を実行し、「Wi-Fi」の「IPv4 アドレス」を見てもOKです。

---

## 2. フロントの API 向き先を `<PC_IP>` に変える

`.env`（プロジェクト直下）の次の行を、`localhost` から **自分の `<PC_IP>`** に変更：

```dotenv
# 変更前
NEXT_PUBLIC_API_BASE_URL=http://localhost:8000/api/v1
# 変更後（例）
NEXT_PUBLIC_API_BASE_URL=http://192.168.128.103:8000/api/v1
```

**なぜ必要？** スマホにとって `localhost` は「スマホ自身」を指してしまい、PC の API に届かないため。

> 変更するのは **`NEXT_PUBLIC_API_BASE_URL` の 1 キーだけ**です。書式は `http://<PC_IP>:8000/api/v1`
> （**`http`／ポート `8000`／末尾 `/api/v1` まで**含める）。`localhost` のまま残さないこと。
> 他のキー（Supabase 関連など）はスマホ検証では変更不要です。

---

## 3. フロント／バックを**リビルド**して反映する

このプロジェクトは**ソースをイメージに焼き込む構成**（`COPY`、ボリュームマウントではない）です。
そのため `.env` の変更や `git pull` の新コードは、**コンテナをリビルドしないと反映されません**。

```bash
docker compose up -d --build frontend backend
```

**リビルドが必要かどうかの早見表**：

| 変更したもの | リビルド |
|---|---|
| `.env`（`NEXT_PUBLIC_*` 等） | **必要**：`docker compose up -d --build frontend backend` |
| `next.config.ts` / フロントのソース | **必要**：`docker compose up -d --build frontend` |
| backend のソース | **必要**：`docker compose up -d --build backend` |
| `docs/` のみ（このマニュアル等） | 不要 |

> 「最新を取り込んだのに古いまま」「.env を変えたのに効かない」はほぼこのリビルド漏れが原因です。

---

## 4. Windows ファイアウォールで 3000 / 8000 を許可する

スマホ → PC の通信は、既定のファイアウォールでブロックされます（特にネットワークが「パブリック」のとき）。
**管理者の PowerShell**（スタート → PowerShell を右クリック →「管理者として実行」）で、同一 LAN 限定の許可ルールを追加：

```powershell
New-NetFirewallRule -DisplayName "Climo Dev 3000 (LAN)" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 3000 -RemoteAddress LocalSubnet -Profile Any
New-NetFirewallRule -DisplayName "Climo Dev 8000 (LAN)" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 8000 -RemoteAddress LocalSubnet -Profile Any
```

- `RemoteAddress LocalSubnet` … 同じ LAN 内からのみ許可（外部には開かない）。
- 検証が終わって元に戻したいときは：
  ```powershell
  Remove-NetFirewallRule -DisplayName "Climo Dev*"
  ```

---

## 5. スマホで開く

スマホのブラウザ（Android Chrome / iOS Safari など）で：

```text
http://<PC_IP>:3000
```

（例：`http://192.168.128.103:3000`）

- ログイン画面 → メール / パスワードでログイン → ホーム表示・コーデ提案まで動けば**成功**です。
- URL バーに「保護されていない通信（⚠）」と出ますが、LAN 内の HTTP 検証では正常です。気にしなくてOK。

---

## 6. うまくいかないときの早見表

| 症状 | 原因 | 対処 |
|---|---|---|
| ページ自体が**開けない**（タイムアウト） | Wi-Fi 不一致 / ファイアウォール未許可 / `<PC_IP>` 間違い | 手順 0・4・1 を再確認 |
| 画面は出るが**「認証状態を確認中…」で固まる** | Next.js の dev リソース cross-origin ブロック（後述） | 手順 7 を確認。通常はリポジトリ設定済みなので**リビルド漏れ**を疑う |
| ログインで **`Invalid login credentials`** | メール / パスワード違い、または未登録 | 正しい資格情報を入力 or 新規登録。アカウント有無は代表者（C）に確認可 |
| 画像が表示されない / API だけ失敗 | `NEXT_PUBLIC_API_BASE_URL` が `localhost` のまま | 手順 2・3 を再確認 |

---

## 7. 補足：「認証状態を確認中…」で固まる現象について（重要な背景）

Next.js 16 は、開発サーバのリソース（JS チャンク・HMR・フォント）への**別オリジンからのアクセスを既定でブロック**します。
スマホは `http://<PC_IP>:3000` という「別オリジン」で開くため、ブロックされるとクライアント JS が配信されず、
**画面の HTML だけ出て JavaScript が動かない**（＝認証チェックが始まらず「確認中…」のまま固まる）状態になります。

**対策はリポジトリに設定済み**です（[`frontend/next.config.ts`](../frontend/next.config.ts)）：

```ts
allowedDevOrigins: ["192.168.*.*", "10.*.*.*", "172.16.*.*", "172.31.*.*"],
```

private IP 帯を許可しているので、**各自が自分の `<PC_IP>` で検証する分には編集不要**です。
それでも固まる場合は、**`docker compose up -d --build frontend`（リビルド）忘れ**がほぼ原因です。

**まずこの 1 コマンド**で現象を確認できます：

```bash
docker compose logs frontend --tail=200 | grep -i "Blocked cross-origin request"
```

この行がヒットすれば本現象です。何も出なければ別原因（手順 6 の早見表へ）。

### セキュリティ上の位置づけ（重要）

- `allowedDevOrigins` は **`next dev`（開発サーバ）でのみ有効**で、本番ビルド（`next build` / `next start`）には一切影響しません。
- 想定利用は **同一 LAN の実機表示確認のみ**。private IP 帯（自宅・社内 LAN）を許可する設計です。
- ⚠ **公衆 Wi-Fi（カフェ等）での実機検証は非推奨**。同一ネットワークの第三者から PC の dev サーバへ到達し得るため、信頼できる LAN で行ってください。

### バージョン依存の注記

- この挙動・設定キー名（`allowedDevOrigins`）は **Next.js 16 系**のものです。将来 Next.js を上げた際は、
  仕様変更でキー名や既定挙動が変わる可能性があります。アップグレード時はこの節の再確認を推奨します。

> 補足：本現象は **Supabase 認証やセキュアコンテキスト（HTTPS 要否）とは無関係**です。
> 原因は Next.js dev の cross-origin ブロックなので、Supabase 周りを調べても解決しません（誤調査回避）。

## 8. 検証が終わったら（PC 単独開発に戻す）

戻し漏れ防止のチェックリスト：

- [ ] `.env` の `NEXT_PUBLIC_API_BASE_URL` を `http://localhost:8000/api/v1` に戻した
- [ ] リビルドした：`docker compose up -d --build frontend`
- [ ] （任意）ファイアウォールのルールを削除した：管理者 PowerShell で `Remove-NetFirewallRule -DisplayName "Climo Dev*"`

---

## まとめ（チートシート）

```text
1. PC_IP を調べる（Get-NetIPAddress / ipconfig）
2. .env: NEXT_PUBLIC_API_BASE_URL = http://<PC_IP>:8000/api/v1
3. docker compose up -d --build frontend backend   ← リビルド必須
4. 管理者PS: 3000/8000 を LocalSubnet で許可
5. スマホ（同一Wi-Fi）で http://<PC_IP>:3000
```
